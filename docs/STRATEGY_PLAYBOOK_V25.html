小杰AI选股系统 Pro V26.5 · 选股战法体系说明

文档定位：基于当前仓库实现（scan_engine.py、pool_manager.py、core/strategies/*）整理的产品级战法说明书，便于理解「每一步在交易上想解决什么问题」，而非 API 手册。
版本：与 constants.APP_VERSION（V26.5）对齐；若本地改过 config.yaml 或策略实验室覆写，请以运行态配置为准。

一、系统全局底座与战略定位

1.1 DuckDB 与「宽表」到底意味着什么？

实盘链路里，DuckDB 是日线与特征的中枢：行情、复权、技术指标、北向与主力等字段在入库时尽量一次算清、增量修补，供 P1 洗盘、scan_engine 全池扫描、回测与巡检共用。

口径上要分清两层说法，避免和旧文档「52 维」打架：

  • 工程宽表：在约 55 个基础行情与指标字段之上，夜间增量管道还会维护 capital_resonance_score（资金共振） 与 fund_memory_score（股性记忆） 两列；常量列表名里仍习惯叫 ALL_55_COLS，属于历史命名习惯。
  • 科研或文档里的「52 维」：多指出离线工具或早期 KPI 语境，与当前实盘宽表不是同一张维度表。读审计底稿时，以 data_fetcher 与 db_core 契约为准。

交易意图：把「算一次、处处用」写进架构，减少 UI 扫描时重复造轮子；代价是表结构演进要克制，大改列集要配合全量重算或迁移。

1.2 市值门槛：「生数据」与「入选池」是两条线

当前常量（constants.py）里有两道容易混淆的闸：

门槛 典型含义（万元口径） 交易意图
DAILY_BASIC_MIN_MV_WAN（默认 100 亿） 日线同步与下载侧：总市值或流通市值任一侧达到该量级，才进入「当日活跃候选」 数据供给侧：先把明显小票噪音从生数据层挡在外面，减轻存储与特征计算；接近「大资金、机构级阵地战」的数据底座
P1_SELECT_MIN_CIRC_MV_WAN（默认 60 亿，可由 config.yaml → strategies.p1.select_min_circ_mv_wan 覆写） P1 候选 SQL、pool_manager 安检、scan_engine 初筛共用的流通市值下限 选股需求侧：在「不要太小」与「不要过度精英化」之间折中；若库里历史行只有 ≥100 亿，把 P1 改成 60 亿也未必立刻多出票，取决于本地库是否真有 60～100 亿区间的日线

一句话：100 亿更像「养数据的护城河」；60 亿（可配）是「底仓池要不要稍微宽一点」的产品旋钮。两者叠加，共同塑造系统的大中盘审美，而不是游资小票擂台。

1.3 「机构级阵地战」在代码里的落点

  • 流动性与体量：流通下限加上日线里的成交额、换手、主力与北向等，让信号默认发生在「能承载大资金」的标的区间。
  • 右侧权重结构上移：守护进程在 14:35 起进入 P4 优先走廊，禁止再派 P3 扫描线程；尾盘 14:35、40、45、50 四枪 P4 子线程内可长达 15 分钟等锁——工程上明说：尾盘确定性比盘中噪音更贵。
  • 分时不是微观博弈引擎：系统依赖 六槽快照（935、1030、1125、1325、1425、1440）把量能与 VR 锚在几个可复现时点，而不是在 UI 里用 Tick 级盘口做高频博弈；P3 的「热成像」也是按代码与表结构读取统计切片，不是炒单逻辑。

二、P1 底仓初筛：十一维平滑分与 pass_line

P1 是全系统的基因层：它决定哪些票「有资格」进入 P2～P5 的战法视野（在 scan_engine 里，P1 平滑分不过线则不再跑后续战法匹配，并会计入 p1_prescreen 统计）。实现上，核心数学全部在 score_calibration.compute_p1_multi_dim_smooth_score，pool_manager 与扫描侧共用同一函数，避免「池子里一套分、扫描里另一套分」。

2.1 商业意图：为什么用「平滑插值」而不是硬阈值瀑布？

A 股日线噪声大，硬 if/else 容易产生「差 0.01 就从满分掉到零分」的悬崖效应。十一维大量使用 np.interp 分段线性映射，把连续基本面与技术面信息压到有上限的贡献，再在总分上 cap 到 100。意图是：包容正常波动，惩罚极端失衡，让池子「有一批合格的中坚力量」，而不是只剩完美图形。

2.2 十一维逐项解读（交易语言）

下列满分与节点以源码为准，这里强调每维在解决什么决策焦虑。

  1. 筹码真空（最高 18 分）　用真实自由换手 turnover_rate_f（缺失时用成交量反算）衡量筹码是否在交换。节点覆盖 0.8%～8% 等日常可达区间。意图：奖励适度活跃、惩罚死筹或过度狂热；当日换手为 0 时用近 5 日均换手兜底，避免停牌或数据缺口把整条维度打穿。
  2. 趋势距离（最高 16 分）　看 MA20 相对 MA60 的百分比距离。意图：多头拉开、趋势有空间时给分；拉得太夸张则降分——避免「趋势末端博最后一棒」的审美。
  3. 均线成熟（最高 12 分）　看 MA20/MA60 比值是否在 1.02～1.15 附近的「成熟多头平台」。意图：偏好走稳的中期结构，而不是均线乱纠缠或过度乖离的畸形态。
  4. 资金攻击（最高 16 分）　双因子：近 5 日时间加权（主力+北向）净流入/成交额映射一部分；再用 max(量比, 换手/2) 作活跃度代理映射另一部分，合计封顶 16。意图：把「钱真的在进」与「今天有人交易」拆开又合并，缓解单一字段量纲不一致导致的全场零分。
  5. 波段涨幅（最高 12 分）　用 max_60d_pct，8%～25% 为核心舒适区（偏大盘主升审美）。意图：既不要死鱼，也不要已经透支完波段、只剩博弈空间的品种。
  6. 黄金起爆（最高 7 分）　涨幅与量比共振：例如涨幅 4%～6% 且量比 1.8～2.5 给满档，弱一档给部分分。意图：捕捉「温和放量突破」的典型机构审美，而不是无脑追涨停。
  7. 趋势健康（最高 4 分）　用 bias20（乖离） 的插值，贴线略优，双向极端降分。意图：价格离均线太近或太远都有问题——太近可能没趋势，太远可能回撤风险大。
  8. 主升斜率（最高 8 分）　ma20_slope_5（五年化斜率百分数）过快或过慢都降分。意图：识别「匀速主升」，过滤躺平和疯狗斜率。
  9. 动态 PE（最高 4 分）　相对行业 75 分位 q75 插值，过高连续降分。意图：同一行业内横向比较，防止估值离谱全靠题材撑；这是软分，不是 PE 硬闸的全部。
  10. 高位熔断（基础 3 分，可倒扣）　bias20 过高时，先压「熔断」维得分，再额外 −12。意图：与 P1 前置乖离类硬闸互补，在平滑总分里再扎一根钉子，减少「妖股靠一两项拉满混进池子」。
  11. 市值与板块加分（附加）　动态行业贝塔；板块排名前 3 名 +5、前 8 名 +2；流通市值（亿）500～1000、1000～2000、2000+ 分档加 2～4 分。意图：同样技术面下，优先「行业在风口、板块顺位靠前、体量够承载」的标的。

2.3 pass_line：及格线哲学（默认约 50 分）

  • 来源：config.yaml → strategies.p1.profiles 下 strict、neutral、relaxed 三档的 pass_line，由 config_manager.get_p1_regime_thresholds 按大盘环境映射；默认多档为 50。
  • 放行含义：十一维（及下文融合）之后 总分 ≥ pass_line 才视为「P1 基因及格」，扫描侧才继续 P2～P5；否则统一理由 「平滑得分不达标」，明细里带 「未达标比对」。
  • 交易意图：50 分是包容性与僵尸过滤的折中——不是「满分才配活着」，而是「多项维度没有结构性烂掉」。可在实验室或 YAML 里收紧到 52～55 提高纯度，或配合观察池逻辑（主池更严、缩量期观察池略宽）。

2.4 股性记忆 fund_memory_score（可选凸组合）

若 fund_memory_weight_p1 > 0（来自 YAML 或常量 FUND_MEMORY_WEIGHT_P1），最终分会在 capped 前做凸组合（记忆分列 0～200 先映射到 0～100）：

最终 ≈ (1 − w) × 十一维小计 + w × 记忆分(0～100)

意图：奖励「近期有资金痕迹、股性活跃」的大票；w 设为 0 则纯十一维，入选更严。P1 不把 capital_resonance_score 当硬闸（与 P3～P5 动态分里的共振注入分离）。

2.5 P1 前置安检（pool_manager 侧，与扫描对齐）

在十一维之前，还有流通市值闸、解禁与惩罚黑名单、PE 与流动性、Regime 阈值（MA120 粘合、MACD 柱、量价背离等）与容错档机制。意图：先挡「明显不能碰」的雷，再谈打分；强信号时适度放宽部分阈值，避免机械规则错杀主升浪——具体见 pool_manager 文档字符串与淘汰表 reason。

三、P2 竞价与 P3 盘中：巡逻、跟踪，而非主决战

3.1 P2：开盘「定调」的四策略胸甲

  • 结构：硬筛选在 p2_auction_screener.py（全局否决与四大竞价策略）；strat_p2_auction.P2Auction 负责组装 burst、surge、风控合并。
  • 黄金门禁：由 scan_engine 统一调用 strict_golden_burst_ok；对核心筛选通过标的可放宽，避免「策略三」这类标签被误杀。
  • 交易意图：竞价是信息密度最高的几分钟，用较少字段判断「今天资金认不认、开盘结构健不健康」，给盘中与尾盘接力做名单，而不是代替 P1 做底仓定义。

3.2 P3：盘中七策略与风控，但战略上已让路尾盘

  • 结构：p3_intraday_screener.py 硬阈值与七大策略；strat_p3_intraday.P3Intraday 负责盘口组装、日内惩罚（如 VWAP、上影、早盘量比）、可选 DuckDB vol_slice_60d Z-Score、爆发分合成。
  • 工程上的「降权」（比单纯 pool_weight 更硬）：守护进程 auto_sniper_daemon 在 09:30～11:30、13:00～14:30 才允许 P3 扫描节拍；14:35 起进入 P4 优先走廊后，P3 不再派线程，避免与尾盘四枪抢 _SCAN_BUSY。P3 仅非阻塞抢锁；尾盘 P4 可阻塞等锁至多 15 分钟。
  • 综合分侧：scan_engine 里 pool_weight_map 对 P3 为 1.0，P2 为 0.92，P4/P5 为 0.95——数字上 P3 并非被乘到最小；真正的降级在调度与业务叙事：系统把算力与确定性押在 14:35～14:50 的 P4，盘中更多是巡逻与接力。

3.3 「清除分时微观态博弈」在实现上指什么？

  • 系统不在 scan_engine 里做 Tick 级盘口博弈链；分时依赖多时点快照槽位写入 JSON，供 VR 与量能锚点使用。
  • P3 的「热成像」来自预聚合切片表的统计矩，而不是秒级逐笔推理。
  • 交易意图：把可复现、可审计的中等频率信息当作主菜；不鼓励用不可回放的微观噪音当核心 alpha。

四、P4 尾盘「盲狙」：14:40～14:50 的右侧核心窗

4.1 时间窗与工程保障

  • 四枪时间：14:35、14:40、14:45、14:50（每枪子线程内最长等锁 900 秒）。
  • 产品话术上的「重仓核心窗」：注释中把 14:40～14:50 标为与 14:35～14:50 走廊对齐的业务重仓区间——前一枪接尾盘前序结构，后三枪落在收盘确定性最高的阶段。
  • 与 P3 的零重叠：14:35 起 P3 已停派，不存在「盘中扫描与尾盘抢锁」的内耗。

4.2 战法层：八策略物理胸甲

硬阈值在 p4_tail_screener.py；引擎在 strat_p4_tail.P4Tail。特点：尾盘量比、成交额相对流通市值、筹码集中度、尾盘异常放量等映射为分项；叠加三层风控与黄金门禁放宽（核心筛选通过时）。交易意图：利用全天信息出齐、真假突破更难伪装的窗口，做右侧确认；适合作为调仓与次日预期的锚点，而不是开盘赌方向。

4.3 在结果表里的「战略地位」

P4/P5 命中多条战法时，scan_engine 会优先保留带「金·」「实锤」「九转序列」「龙虎榜机构」「布林CCI」「融资杠杆主力」「主力真金」「机构实锤」「★」等标签的说明，避免一行里堆七八个名字却不知道主逻辑是什么。交易意图：尾盘信号要可复盘、主叙事单一，方便次日执行与风控。

五、P5 盘后真龙与次日 09:35 二次验证

5.1 P5：收盘后的「全量定档」

结构：p5_postmarket_screener.py 向量化硬逻辑；strat_p5_postmarket.P5Postmarket 单股窗口拼接与打分。特色软约束：例如 MA5>MA20 且乖离过大时的虚火降权（按流通市值分档乘子）；MA20 五日斜率加速向上时的奖励乘子——意图：盘后看得更清，惩罚假突破、奖励趋势加速，但不一刀切否决。

5.2 次日 09:35：容错闭环（跨进程 JSON）

  • 前夜：P5 会话写入 data/runtime/state/p5_last_session.json。
  • 次日早盘：守护进程在 09:35 快照后跑 early_morning_p5_validation，结果写入 p5_yesterday_validated.json。
  • 企微闸：notification_gateway 对验证结果为「已剔除」的代码跳过推送，减少「昨晚兴奋、今早证伪」的噪音。
  • 交易意图：把 P5 从「一夜情信号」改成可验证的隔夜计划——允许错，但不允许错得无声无息。

六、综合分与共振：P3～P5 的「第二轨」

动态合成里，战法爆发分经 burst_soft_cap、compress_surge_bonus、截面极端抑制 dampen_burst_by_extremes，避免流动性噪声与妖股虚火叠乘。capital_resonance_score 在 P3～P5 路径以 CAPITAL_RESONANCE_WEIGHT_P345（默认约 8%）线性注入；P1 排序若用共振键则另有 CAPITAL_RESONANCE_WEIGHT_P1（约 18%）的语境——二者刻意分离，防止同一列在全链重复刷存在感。

七、优点与局限（诚实清单）

优点

  • 分层清晰：P1 解决「谁配进战场」，P2/P3 解决「何时试探」，P4 解决「收盘前下注」，P5 与次日验证解决「隔夜可信度」。
  • 工程与交易对齐：尾盘优先走廊、等锁策略，把稀缺资源押在信息最充分的时段。
  • 平滑分与及格线：比纯硬闸更少「一票否决误杀」，比纯排序更可解释（score_details 可展开到 UI）。
  • 特征管道化：共振与股性记忆夜间修补，白天扫描少拉 API、多信库。

局限与偏差来源

  • 数据依赖：生数据 100 亿门槛下，若历史从未落过 60～100 亿区间，调低 P1 下限也不会变魔术。
  • Regime 与实验室覆写：pass_line 与阈值若与默认值偏离，本页描述需以当前 YAML 为准。
  • P1 与 Streamlit 基因缓存：strat_base.get_cached_p1_gene 在带 UI 会话时读缓存；CLI 或无 session 路径有回退逻辑，极端情况下分与 UI 不完全一致——应用层应以 scan_engine / pool_manager 实际写入的 p1_score 为准。
  • 仍非圣杯：十一维是风格化筛选，在单边熊市或结构分化极端时，及格池可能变薄；需结合仓位与手动黑名单。

八、使用注意事项（运维与实盘）

  1. 先数据，后战法：新环境先保证 sync_recent_days、_sync_daily_features() 把共振与记忆列补齐，再谈 P1 池规模。
  2. 统一版本与配置：改 constants 或 config.yaml 后，重启 Streamlit 与守护进程，避免一半进程读旧缓存。
  3. 不要双进程抢队列：异步扫描队列由 filelock 互斥；不要同时开两个消费者进程抢 pending。
  4. P4 时间窗别手动堆任务：尾盘 15 分钟等锁是为挤进扫描；若同一时段跑重型 SQL，仍会拖长队列。
  5. 观察 p1_prescreen 与淘汰表：池子变少时，先看是市值闸、平滑分还是前置安检，不要只盯战法层。
  6. P5 验证 JSON 权限：data/runtime/state/ 需进程可写，否则次日验证链断裂、企微闸失效。
  7. 全量代码底稿：可执行 python tools/generate_system_full_snapshot.py 再生成 HTML 快照。
